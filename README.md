# 額装シミュレーター（gakuso-simulator）

コーナーサンプル画像と作品画像から額装完成イメージをCanvas合成する店頭・工房向けシミュレーター。単一HTMLファイル・完全クライアントサイド。

- 業務仕様の正本: [`要件定義書_v24_画像前処理・エクスポート.md`](要件定義書_v24_画像前処理・エクスポート.md)
- 実装ガイド: [`CLAUDE.md`](CLAUDE.md)
- 共通標準: 社内の開発標準に準拠（GAS固有条項は適用外。設計思想のみ適用）

## 使い方

**公開URL: https://shuzo-yamamoto.github.io/gakuso-simulator/**

上記URLをブラウザで開けばそのまま使えます（PC: Chrome/Edge、iPad: Safari）。
インストール・サーバー・ビルドは不要です。

### オフラインで使う場合

リポジトリから `index.html` を1ファイルだけ取得すれば、単体でそのまま動作します。

1. リポジトリを取得（またはZIP展開）
2. `index.html` をブラウザで開く

## Step n 完了確認手順
（各Step完了時に、実装AIがここへ確認手順を追記する — CLAUDE.mdの規約）

### Step 1 — 無地モード（処理B: プロファイル押し出し）

#### 手動での動作確認（T-1）

1. `index.html` をブラウザで開く
2. 作品画像に任意の画像、コーナーサンプルに `test-images/IMG_3569.jpg` を選択
3. 黒いフレームの**上辺**を2点クリックで囲む（銀の縁を含め、角には掛からない範囲）
4. 「フレームの種類」で **無地（単色・シンプル）** を選ぶ
5. 期待結果
   - 「切り出したテクスチャ」プレビューが、押し出し後（長手方向に一様）の見た目に変わる
   - 「テクスチャスケール」スライダーがグレーアウトし、注意書きが出る
   - プレビューのフレームに明暗の縞・タイルの継ぎ目が見えない
   - 「柄物」に戻すと即座に元の見た目に戻る（フレーム幅の値は変わらない）
6. ブラウザのコンソール（F12 → Console）で `checkStripUniformity()` を実行 → **ΔY ≤ 5**（無地モードでは 0 になる）
   - 同じ帯を「柄物」にして実行すると 19 前後になり、改善量を数値で比較できる

#### 後方互換の確認（T-2）

柄物モード・テクスチャスケール100%で、v23と描画結果が画素レベルで一致することを確認する。

1. 比較用に v23 のファイルを取り出す: `git show v23:index.html > v23.html`
2. `index.html` と `v23.html` を同一のローカルサーバー配下に置き、それぞれで同じ操作を行う
   - クリック位置のブレをなくすため、両方のコンソールで座標を直接指定する:
     ```js
     pts=[{x:100,y:20},{x:790,y:290}]; buildTex(); drawPreview();
     document.getElementById('preview').toDataURL().length
     ```
3. 両者の `toDataURL()` の文字列が完全一致すれば合格

> **注意**: 油彩額の「ライナー」だけは、麻地テクスチャに `Math.random()` のノイズが入る v23 由来の仕様のため、
> 同じ設定でも描画のたびに画素が変わる。比較する場合は両方で `Math.random` を固定の疑似乱数に差し替えてから実行すること。

### Step 2 — 照明ムラ補正（処理A）＋ミラータイリング（処理C）

#### 手動での動作確認（T-3・T-4）

1. コーナーサンプルに `test-images/IMG_3569.jpg` を選び、黒いフレームの上辺を2点クリックで囲む
2. 「フレームの種類」は **柄物** のまま（＝v23と同じ貼り方）
3. 「照明ムラ補正をかける」のON/OFFを切り替える
   - ONにすると「切り出したテクスチャ」プレビューの明るさのかたよりがならされる
   - OFFにすると「補正強度」スライダーが消える
4. コンソールで `checkStripUniformity()` を実行し、ON/OFFでΔYを比べる
   - **合格基準（T-3）: 補正ONのΔYが、補正OFFのΔYの 1/3 以下**
5. コーナーサンプルを `test-images/frame1.jpg` に替え、補正ON/OFFで彫刻パターンの山の数・位置が変わらないことを目視で確認（**T-4**）

#### ミラータイリングの確認（T-6）

1. 柄物モードで「タイルを1枚おきに反転して貼る」をON/OFFして見比べる
2. ONのときフレームの模様が鏡合わせになり、タイルの継ぎ目が見えなくなること
3. 数値で確認する場合は、コンソールで以下を実行する（製品コードには測定関数を持たせていないため、その場で貼り付ける）

   ```js
   // タイル境界の直前列と直後列の輝度差の最大値を返す（合格基準: ≤ 2）
   (function(){
     const s=lastHStrip, w=s.width, h=s.height;
     const d=s.getContext('2d').getImageData(0,0,w,h).data;
     const tsPct = frameKind==='plain' ? 1 : (parseInt(document.getElementById('ts').value)||100)/100;
     const step = Math.max(4, Math.round(tileTopBot.width * tsPct));
     const Y=(x,y)=>{const p=(y*w+x)*4; return 0.299*d[p]+0.587*d[p+1]+0.114*d[p+2];};
     let max=0;
     for(let x0=step; x0<w; x0+=step)
       for(let y=0;y<h;y++) max=Math.max(max, Math.abs(Y(x0-1,y)-Y(x0,y)));
     return max;
   })()
   ```

#### 補正強度の確認（T-5）

「補正強度」を **0%** にしたときの描画結果が、「照明ムラ補正」OFF時と画素一致すること。
コンソールで両方の `document.getElementById('preview').toDataURL()` を取り、文字列比較で確認する。

#### 後方互換の確認（T-2）

Step 1 の手順と同じ。ただし **v24側は「柄物・照明ムラ補正OFF・ミラータイリングOFF・スケール100%」** に設定すること。
ミラーONではタイルの並べ方が変わるため、v23とは一致しない（要件定義書§3.3「後方互換との関係」）。

### Step 4 — エクスポート（処理E）

#### 手動での動作確認（T-8・T-9）

1. シミュレーションを実行し、結果が表示された状態にする（表示前は保存ボタンが押せない）
2. 「設定情報を含める」を**ON**にして「📥 画像を保存（PNG）」を押す（**T-8**）
   - ボタンが「生成中…」になり、完了後に元に戻ること
   - 保存された画像を開き、次を確認する
     - 額装部分の解像度がプレビューの**縦横2倍**
     - 下部に白い帯があり、日時・額縁種類・作品サイズ・マット/面金（または ライナー/入子）の設定が読めること
     - 帯の最下行に注意書き（免責文言）が入っていること
     - ファイル名が `額装シミュレーション_297x210mm_20260720-143025.png` の形式であること
3. 「設定情報を含める」を**OFF**にして保存する（**T-9**）
   - 白帯がなく、画像の高さがプレビューのちょうど2倍であること
   - 画像の**右下**に注意書きが薄く（半透明で）焼き込まれていること

#### プレビューとの一致確認（T-10）

コンソールで以下を実行すると、寸法がすべて厳密に2倍になっているかを一括で確認できる。

```js
// 全項目 true / mismatches が空なら合格
(function(){
  const L1=computeLayout(1), L2=computeLayout(2);
  const bad=Object.keys(L1).filter(k=>k!=='S'&&k!=='tsPct'&&L2[k]!==L1[k]*2);
  return {mismatches:bad, ok:bad.length===0, L1, L2};
})()
```

目視では、保存した画像とプレビューを並べて、フレーム・マット・面金・ライナー・作品の**配置と模様の繰り返し数**が一致していることを確認する。

#### iPad Safari での確認（重要）

iPad では `download` 属性が効かないため、**新しいタブで画像が開き「長押しで保存してください」の案内が出る**動作になる。

- 案内が表示されること
- 画像を長押しして「”写真”に保存」で保存できること
- ポップアップブロックの警告が出た場合は、ブラウザ設定で許可してから再実行すること

### 設計判断ログ（仕様書に無い実装判断）
<!-- 要件定義書に明記が無く実装上決めた事項を、理由とセットで残す。後から追えるように -->
- v23以前はチャットベースで反復開発しており、v23コードが設計を兼ねていた。v24からGitHub管理＋設計書ファーストに移行（2026-07-19）
- **Step 1（2026-07-20）**: 要件定義書§3.2-3「プロファイル列を長手方向に必要長さぶん引き延ばす」の実装方法として、
  1画素幅の列を **元の切り出し帯と同じ sw×sh に複製して** `tileTopBot` / `tileSide` に渡す方式を採った。
  理由: サイズを元と同じに保てば、転置処理・`buildLongStrip()`・フレーム幅の自動設定を v23 のまま流用でき、
  柄物モードの後方互換（T-2）を壊すリスクが最小になるため。複製は `drawImage` の拡大ではなく画素の直接コピーで行う（補間による滲みを避け ΔY=0 を保証するため）
- **Step 1（2026-07-20）**: 種類切替時の `buildTex()` 再実行では、フレーム幅の自動設定をしない（`buildTex(false)`）。
  利用者が調整済みのフレーム幅を切替のたびに上書きしてしまうのを避けるため。初回の切り出し時は従来どおり自動設定する
- **Step 1（2026-07-20）**: 無地モードでは「テクスチャスケール」をグレーアウトするだけでなく、内部的にも `tileScale=1` 固定とした（要件定義書§3.2-4の趣旨。押し出しに繰り返し周期が存在しないため）
- **Step 2（2026-07-20）**: 要件定義書§3.3（処理C）と§7 T-2（後方互換）が両立しない矛盾を発見。
  `buildLongStrip()` は `count = ceil(targetLen/step)+1` で必ず2枚以上タイルを並べるため、スケール100%でも i=1 のタイルが存在し、
  ミラーONでは柄物モードでも v23 と一致しない。**ミラータイリングON/OFF切替UIを設け、T-2はミラーOFFで判定する**方針で設計書を更新して解決
  （要件定義書§3.3「後方互換との関係」・§4.1・§7 T-2・§9 確定済みNo.1）
- **Step 2（2026-07-20）**: ボックスぼかしの偶数サイズの扱い。SVG仕様は「大きさ d のぼかしを画素境界を中心に2回」と記述しているが、
  実装では**1回目と2回目で左右の広がりを逆にする**（例 d=4 なら 左2右1 → 左1右2）ことで等価とした。3回目は大きさ d+1（奇数＝左右対称）。
  移動和の実装で素直に書けるため
- **Step 2（2026-07-20）**: 照明ムラ補正OFF・k=0 のときは `applyFlatField()` を**呼ばない**（短絡）。
  §3.1-5 の式に k=0 を入れれば数学的には元画素に戻るが、計算を通さないことで丸め誤差の余地をなくし T-2・T-5 を確実にするため
- **Step 2（2026-07-20）**: ミラータイリングの切替は `drawPreview()` の再実行のみで反映する（`buildTex()` を呼ばない）。
  タイルの並べ方だけが変わり、切り出し帯そのものは変わらないため
- **Step 4（2026-07-20）**: 描画の共通関数化は `computeLayout(S)`（寸法計算）と `renderFraming(ctx, L)`（描画）に分けた。
  `drawPreview()` は `computeLayout(1)` を渡す薄い入口として残し、**外部インターフェース（引数なし・戻り値なし）を変えていない**。
  これにより、スライダー等の既存呼び出し10箇所以上に一切手を入れずに済んだ
- **Step 4（2026-07-20）**: スケール適用で S 倍を忘れやすい箇所が3つあり、いずれもT-10を静かに壊す。実装時は要注意。
  ① 面金の幅 `MK` とライナーの幅 `LW`（v23は表示倍率 `sc` を掛けていないため、`sc` の掛け忘れと混同しやすい）
  ② `buildLongStrip()` のタイル繰り返し単位 `step`（S倍しないと辺の長さだけ倍になり、模様の繰り返し数が倍増して半分のサイズに見える）
  ③ 麻地テクスチャのタイル（S倍しないと織り目だけ細かいままになる）
- **Step 4（2026-07-20）**: 麻地テクスチャの `Math.random()` を固定シードの疑似乱数に置き換えた（要件定義書§3.5-1）。
  プレビューとエクスポートで織り目が一致しないとT-10を満たせないため。
  副次的にアプリ全体が決定論的になり、回帰テストで `Math.random` を差し替える必要がなくなった。
  ただしライナー構成は v23 と画素一致しなくなるため、§7.2.1 で T-2 の判定対象外とした
- **Step 4（2026-07-20）**: フォールバック用の新規タブは、クリック直後の同期処理で開く。
  `canvas.toBlob()` のコールバック内で `window.open()` を呼ぶとポップアップブロックされるため（要件定義書§3.5-5）

### 次Stepへの申し送り
- **Step 3（処理D: 4点パース補正）の差し込み位置**: `buildTex()` の切り出し（`raw` の生成）そのものを射影変換に差し替える。
  要件定義書§3.4-2のとおり、矯正後の長方形が処理A〜Cの入力になるため、処理Aより前に入る。
  なお `computeLayout()` / `renderFraming()` には手を入れる必要がない（切り出し帯より下流は変わらないため）
- **補正強度の既定値**（要件定義書§9 残No.2）: 実測では画像②で k=100 のとき ΔY 19.09 → 4.33（1/4.4）に改善し、模様の破綻も見られない。
  実機確認で強すぎる感触がなければ既定100%のままで確定してよい
- **エクスポート倍率を変えたくなった場合**: `EXPORT_SCALE` 定数（既定2）を変えるだけで対応できる。
  ただし要件定義書§3.5-1・§7 T-8 が「2倍」を前提にしているため、変更するなら先に設計書を更新すること
- **iPadでのフォールバック動作は未検証**: PCではダウンロード経路（`<a download>`）のみ確認済み。
  新規タブ＋長押し案内の経路は実機確認が必要（README「Step 4 完了確認手順」参照）
